# 组件入库标准

这个仓库不是链接收藏夹，而是面向真实项目拼装的组件目录。一个组件进入目录前，需要能帮助开发者做出可靠的选型判断。

## 可以加入的情况

- 有公开 GitHub 仓库。
- 许可证清楚。
- 能解决一个明确的模块级问题。
- 文档说明了安装、配置或接入方式。
- 可以独立接入，或者已经明确说明它会带来的架构绑定。
- 项目仍在维护，或者足够稳定，低频维护也可以接受。
- 有可信的采用信号、生态使用场景或技术质量。

## 需要谨慎标注的情况

- 许可证是 source-available，而不是标准开源许可证。
- 托管产品成熟，但自托管路径有限。
- 项目要求团队接受较重的平台绑定。
- 组件横跨多个分类，可能形成架构锁定。
- 文档适合 Demo，但生产运维说明不足。

## 应该拒绝或暂缓的情况

- 许可证不清楚。
- 仓库没有有效文档。
- 项目已经停止维护，并且没有稳定到“可长期冻结使用”的理由。
- 只是另一个项目的薄封装 Demo。
- 不能独立接入。
- 与已有条目重复，且没有明确差异。

## 评分规则

使用 1 到 5 分：

- `5/5`：该分类下的强默认选择。
- `4/5`：好选择，但有明确取舍。
- `3/5`：适合特定场景。
- `2/5`：实验性、偏小众或接入成本较高。
- `1/5`：仅作参考，新项目应避免优先采用。

评分是选型辅助，不是绝对质量结论。

## 成熟开源候选筛选

`tools/search_catalog.py --mature-only` 用于快速筛出优先评估的候选组件。一个组件要进入这个结果，至少需要满足：

- GitHub 字段指向公开 GitHub 仓库。
- 许可证字段清楚，且没有明显 source-available、商业许可证或非标准开放限制关键词。
- 评分不低于 `4/5`。

这个筛选只用于缩小候选范围，不等同于正式选型结论。进入项目技术栈前，仍然要人工确认许可证义务、维护状态、文档质量、托管方式、数据边界和团队接入成本。

## 接入成本

- `低`：可以加入项目，对架构影响有限。
- `中`：需要认真处理配置、托管、数据库、工作流或团队约定。
- `高`：会改变架构、运维、身份模型、数据模型或部署策略。

## 必填字段

每个组件条目都应该包含：

- GitHub
- 官网或文档
- 模块
- 技术栈
- 许可证
- 适合
- 不适合
- 接入成本
- 替代方案
- 评分
- 备注
