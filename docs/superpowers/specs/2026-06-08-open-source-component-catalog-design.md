# 开源组件库设计规格

## 目标

建立一个适合发布到 GitHub 的开源组件目录仓库，用于按模块整理可复用组件。开发新项目时，可以从前端、后端、认证、数据库、AI、支付、部署、监控等分类中选择组件，并组合成项目技术栈。

## 范围

第一版是“精选索引库”，不是源码大仓库。仓库保存选型记录、链接、接入说明和模块分类。可运行代码模板以后可以放到 `components/`，但必须等目录格式先稳定并验证有用。

## 架构

仓库按产品能力组织：

- `README.md` 是入口，解释仓库用途和拼装流程。
- `catalog/*.md` 保存按分类整理的组件条目。
- `docs/selection-criteria.md` 定义组件入库标准。
- `docs/project-assembly-guide.md` 说明如何把组件组合成项目技术栈。
- `components/README.md` 预留未来可运行示例和 starter 模板的位置。

这种结构保持简单，不需要安装工具或理解自定义数据库，就能按能力快速扫描候选组件。

## 组件数据模型

每个组件条目记录：

- GitHub 仓库地址
- 官网或文档地址
- 模块分类
- 主要技术栈
- 许可证
- 最适合的使用场景
- 不适合的使用场景
- 接入成本
- 替代方案
- 评分
- 实际选型备注

目录刻意不手写 star 数这类快速变化的指标。后续如果需要，可以用自动化脚本生成。

## 初始分类

第一版包含这些分类：

- 前端
- 后端
- 认证
- 数据库
- AI
- 支付
- 部署
- 监控

每个分类先放入少量已知开源项目。第一阶段的重点是建立可维护结构，再逐步扩充覆盖面。

## 入库策略

组件可以加入目录的条件：

- 有公开 GitHub 仓库。
- 许可证清楚，或明确标注 source-available 等限制。
- 有说明安装或接入方式的文档。
- 能解决明确的模块级问题。
- 有足够的采用或维护信号，值得进一步评估。

许可证不清楚、长期无人维护、只有玩具 Demo、无法独立接入的项目，应该拒绝或谨慎标注。

## 项目拼装流程

新项目先写模块地图。每个能力选择 1 个主组件和 1 个备选组件，再检查许可证、托管要求、数据归属、认证边界和接入成本。最终技术栈应该先记录下来，再开始实现。

## 风险

主要风险是目录腐化。为降低这个风险，条目避免手写易过期数字，重点记录更稳定的选型信息。另一个风险是误以为组件可以无成本拼装。拼装指南要求在确认技术栈前检查兼容性和集成边界。

## 成功标准

仓库达到可用状态时，开发者应该能够：

1. 打开 README 并理解目录怎么用。
2. 按能力找到候选组件。
3. 比较主方案和替代方案。
4. 从多个模块组合出项目技术栈。
5. 用统一格式新增组件条目。
