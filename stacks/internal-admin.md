# 内部管理后台组合蓝图

这个蓝图适合内部仪表盘、CRUD 工具、运营控制台和后台系统。

## 模块地图

| 能力 | 主组件 | 备选组件 | 选择理由 | 主要风险 |
| --- | --- | --- | --- | --- |
| UI | Ant Design | shadcn/ui | 表格、表单、筛选和后台组件成熟 | 面向客户时视觉风格可能偏通用 |
| 前端应用 | Next.js | Vite + React | 路由和部署约定成熟 | 对简单后台可能过重 |
| 后端 | Django | NestJS | 内置 Admin、ORM、认证基础和 CRUD 生产力 | 如果组织只用 TypeScript，会有栈差异 |
| 认证 | Keycloak | Auth.js | 适合组织登录和 SSO | 比应用级认证更重 |
| 数据库 | PostgreSQL | MySQL | 适合保存运营型关系数据 | 需要备份和迁移纪律 |
| 部署 | Dokku | Coolify | 小型内部应用部署简单 | 不是完整多节点平台 |
| 监控 | Grafana + Prometheus | 托管监控平台 | 运维指标和仪表盘清楚 | Grafana 的 AGPL 许可证需要审查 |

## 推荐优先验证

先验证身份和权限，再做 CRUD 页面。很多内部后台的问题都来自访问控制加得太晚。

## 适合

- 运营仪表盘
- 客服工具
- 财务或后台审核工具
- 现有产品的管理面板

## 不适合

- 面向公众的消费者应用
- 高品牌表达的营销体验
- 不需要 Web UI 的一次性脚本
