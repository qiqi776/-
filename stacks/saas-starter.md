# SaaS 起步项目组合蓝图

这个蓝图适合典型 SaaS 产品：账号系统、Web 应用、后端 API、数据库、支付、账单与发票、产品分析、功能开关、部署和基础监控。

## 模块地图

| 能力 | 主组件 | 备选组件 | 选择理由 | 主要风险 |
| --- | --- | --- | --- | --- |
| 前端 | Next.js | Remix | React 应用基础强，路由和部署约定成熟 | 对小型静态站点可能过重 |
| UI | shadcn/ui | Ant Design | 组件可定制，源码归项目所有 | 团队需要有设计约束能力 |
| 后端 API | NestJS | FastAPI | 适合较大 TypeScript 产品后端 | 比轻量 API 有更多框架仪式 |
| 认证 | Supabase Auth | Auth.js | 当 Postgres 可接受时，账号系统搭建很快 | 会和 Supabase 平台产生绑定 |
| 数据库 | PostgreSQL | MySQL | 通用关系型数据库强默认选择 | 自托管时需要负责运维 |
| ORM | Prisma | Drizzle ORM | TypeScript 开发体验好 | 迁移流程会成为架构的一部分 |
| 支付 | Stripe Node SDK | Medusa | 直接接入结账、收款和支付供应商 API | 商业供应商依赖和地区限制 |
| 账单与发票 | Lago | OpenMeter | 覆盖订阅账单、用量计费和发票流程 | 许可证、财务口径和计费事件一致性需要审查 |
| 产品分析 | PostHog | Plausible Analytics | 事件、漏斗和实验能力覆盖 SaaS 增长闭环 | open core 功能边界和分析基础设施需要审查 |
| 功能开关 | GrowthBook | Flagsmith | 功能发布、实验和数据分析能形成闭环 | 需要稳定事件数据和灰度治理流程 |
| 部署 | Coolify | Dokku | 自托管应用和服务管理界面完整 | 仍然需要承担服务器运维 |
| 监控 | Prometheus + Grafana | 托管监控平台 | 指标和仪表盘路线成熟 | 会增加运维开销 |

## 推荐优先验证

先验证认证、数据库、支付、账单与发票、关键产品事件和功能开关能否协同，再做完整 UI。这些模块决定账号归属、计费身份、数据边界、增长指标口径和灰度发布能力。

## 适合

- B2B SaaS
- 订阅制产品
- 可能从内部工具转向客户产品的系统
- 需要快速起步但仍希望可维护的创业产品

## 不适合

- 静态营销站
- 必须避免商业支付供应商的应用
- 第一天就需要复杂企业 IAM 的产品
